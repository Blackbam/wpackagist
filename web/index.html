WordPress Packagist

Manage your plugins with Composer

This site mirrors the WordPress plugin directory as a Composer repository.

How do I use it?

  1. Add the repository to your composer.json
  2. Add the desired plugins to your requirements using wpackagist as the vendor name.

Example

"name": "acme/brilliant-wordpress-site",
"description": "My brilliant WordPress site",

"repositories":[
    {
        "type":"composer",
        "url":"http://wpackagist.org"
    }
],

"require": {
    "aws/aws-sdk-php":"*",

    "wpackagist/advanced-custom-fields":"*",
    "wpackagist/posts-to-posts":"1.4.x"

},
"autoload": {
    "psr-0": {
        "Acme": "src/"
    }

This example composer.json file adds the Wpackagist repository and includes the latest version of Advanced Custom Fields and version 1.4 of Posts to Posts, along with the Amazon Web Services SDK from the main Packagist repository.

Why use Composer?

  • Avoid checking plugins into source control.
  • Avoid having to use git submodules.
  • Manage dependencies between plugins (requires plugins to declare their own composer.json).
  • Manage WordPress and non-WordPress project libraries with the same tools.

How does it work?

  • Periodically scans the WordPress Subversion repository for changes.
  • Fetches the tags for each updated plugin and maps those to versions.
  • Adds trunk as a dev version.
  • Rebuilds the composer package JSON files.

Known issues

  • Version strings which Composer cannot parse are ignored. All plugins have at least the trunk build available.
  • Plugins with unicode names might not work properly.
